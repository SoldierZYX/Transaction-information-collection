# ADR-0001：MVP 架构与边界

状态：待审阅  
日期：2026-07-12

## 背景

系统在每个 A 股交易日执行盘前与盘后两类研究工作流：采集可追溯的公开信息和行情，经过规则计算后生成报告，并通过邮件投递。MVP 必须优先保证可追溯性、可复现性、合规性与故障降级。

## 决策

1. MVP 使用 Python 单体应用与 SQLite 单实例数据库。
2. 业务日期及报告使用 `Asia/Shanghai`；数据库中的时间统一存储为 UTC。
3. 采用分层模块边界：`collectors`、`processors`、`analytics`、`ai`、`reports`、`delivery`、`workflows`。
4. 原始数据先落库，再标准化、去重、关联事件并进行分析。
5. 数据源通过准入调查与人工审阅后才可实现适配器；未批准的来源不可进入生产工作流。
6. AI 为可关闭的辅助层，只接收已筛选证据；输出必须经结构化校验，且不得改变规则计算结果。
7. 报告与邮件投递独立保存状态；邮件失败不回滚数据和报告。
8. 工作流以唯一运行键和任务锁保证幂等；强制重跑必须保留审计记录。

## 模块职责

| 模块 | 职责 | 禁止事项 |
|---|---|---|
| collectors | 合法获取、解析并记录原始数据及来源状态 | 评分、推断投资结论 |
| processors | 字段标准化、去重、事件归并、证据关联 | 直接发送报告 |
| analytics | 股票池过滤、技术指标、确定性评分、盘后验证 | 伪造缺失数据 |
| ai | 受证据约束的摘要、分类和解释 | 新增无证据事实、修改评分 |
| reports | 生成 Markdown 与 HTML 报告 | 直接抓取或计算行情 |
| delivery | SMTP 投递、去重、有限重试 | 修改报告内容 |
| workflows | 编排、锁、运行审计、降级 | 绕过模块边界 |

## MVP 明确边界

- 包含：交易日历、证券基础信息、日线行情、有限的公告/政策元数据、规则评分、盘前盘后报告、可选 AI、邮件投递和审计。
- 不包含：自动交易、券商接口、实时盘中监控、网页界面、PDF 附件、历史回测、云部署、未获准数据源。
- 公告与新闻在 MVP 中仅保存元数据、短摘要和链接；不批量保存受版权保护的全文。

## 工作流原则

盘前在 08:45 开始，08:55 截止数据，目标 09:00 前发送；盘后在 17:00 开始，使用 16:55 前可用的收盘数据。关键行情、证券映射或交易日历不可用时，报告必须标明不可生成正式观察标的，不能补造数据。

## 后果

- 有利于早期在本地以低成本运行，并保留迁移到 PostgreSQL 的路径。
- 数据源接入速度会受合规调查限制；这是刻意的质量门槛。
- AI 故障时可退回规则驱动的基础报告。
